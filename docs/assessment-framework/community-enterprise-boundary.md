# Community and Enterprise Boundary

Editions are separated by **audience and governance**, not merely by roadmap
phase.

## Principle

> Community helps teams understand and improve software.  
> Enterprise helps leaders understand, govern, and modernize an engineering organization.

## Community

- repository assessment
- local execution
- local knowledge graphs
- Shared Rule Platform
- meaningful starter rule packs (future)
- local findings and recommendations
- repository-level scores (future)
- repository-level modernization guidance
- local Engineering Assessment report
- local MCP
- local configuration
- rule authoring and extensibility

## Enterprise

- Enterprise Knowledge Graph enrichment
- cross-repository / application / portfolio assessment
- business capability impact
- ownership and team context
- enterprise standards governance
- centralized / managed suppressions
- advanced proprietary rule packs
- portfolio scoring and comparative benchmarking
- modernization portfolio and waves
- centralized report management
- organization workflows
- RBAC, SSO, audit, and policy controls

## Shared methodology

Both editions use the same dimensions, taxonomy, evidence, severity, and
traceability rules. Enterprise adds **declared** organizational context; it does
not replace repository observation.

## Non-goals of this boundary doc

- Do not split repositories or rename packages/CLI in this milestone.
- Do not make Enterprise Knowledge Graph mandatory for Community assessment.
- Do not gate Community value solely behind Enterprise features.
